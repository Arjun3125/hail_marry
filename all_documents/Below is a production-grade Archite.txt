# Architecture

**Project:** VidyaOS – AI Infrastructure for Educational Institutions
**Version:** v0.1 (Pilot, Cloud-Ready Design)
**Deployment Model:** Hybrid (Cloud Control Plane + Local AI Data Plane)

---

# 1. Architectural Philosophy

VidyaOS is designed using the following principles:

1. Multi-tenant by default
2. Separation of concerns (ERP vs AI)
3. AI as stateless inference service
4. Cloud migration readiness from day one
5. Horizontal scalability of AI layer
6. Deterministic operational core, probabilistic AI layer

The system is divided into:

* Control Plane (Cloud)
* Data Plane (AI Inference Node)
* Client Layer (Web / Android)

## 3.2 API Gateway Layer (Cloud)

Responsibilities:

* TLS termination
* Request validation
* JWT verification
* Role enforcement
* Tenant routing
* Rate limiting
* API logging

Recommended stack:

* Nginx
* FastAPI
* Redis (throttling + caching)

No business logic here. Only routing and enforcement.

---

## 3.3 Authentication & Authorization

### Authentication

* Google OAuth 2.0
* JWT issued after login

### Authorization

Role-based access control:

| Role    | Permissions                      |
| ------- | -------------------------------- |
| Student | View own data, AI queries        |
| Teacher | Manage assignments, upload notes |
| Admin   | Full tenant control              |

Tenant isolation enforced at:

* Database level
* AI namespace level
* API routing level

---

## Database Model

Single PostgreSQL instance with strict multi-tenant isolation:

* All tables include `tenant_id`
* Row-level filtering
* Indexed by tenant + user_id
* Audit logging enabled

ERP database must NEVER be modified by AI.

AI is read-only regarding academic records.

---

# 5. AI System Architecture

AI runs as independent microservice.

## 5.1 Core Components

### 5.1.1 Ingestion Service

Handles:

* PDF parsing
* Lecture transcript extraction
* Chunk segmentation
* Metadata tagging

Hierarchical chunking strategy:

* Document → Section → Subsection → Paragraph

---

### 5.1.2 Embedding Engine

* Uses local embedding model
* Generates vector representation
* Stores in tenant-specific namespace

Each vector record contains:

* tenant_id
* document_id
* chunk_id
* page_number
* section_title
* subject
* embedding_vector

---

### 5.1.3 Vector Database

Options:

* FAISS (local)
* Chroma (local or server)
* Future: Distributed vector DB

Each tenant has isolated namespace.

No cross-tenant retrieval allowed.

---

### 5.1.4 Retrieval Pipeline

```
Query → Embed → Vector Search (Top-K)
      → Rerank
      → Context Deduplication
      → Context Compression
```

Top-K typically 6–10.

Reranking improves precision.

---

### 5.1.5 LLM Inference Layer

Runs on local GPU.

Responsibilities:

* Structured prompting
* Citation enforcement
* Output validation
* Token optimization

Strict prompt templates per task:

* Q&A
* Summary
* Quiz
* Concept map
* Study guide

---

### 5.1.6 Response Formatter

Adds:

* Citation tags
* Source references
* Structured JSON when required

Example:

```
Answer text... [Biology_Chapter3_p12]
```

# 6. AI Task Engine

AI supports multiple modes:

| Mode                | Description                  |
| ------------------- | ---------------------------- |
| Q&A                 | Grounded document answering  |
| Study Guide         | Structured topic summary     |
| Quiz                | Question generation          |
| Concept Map         | Structured graph output      |
| Weak Topic Analysis | Performance-based suggestion |

Task engine selects prompt template dynamically.

# 7. Infrastructure Deployment (Pilot)

## 7.1 Cloud Control Plane

* VPS (₹500–₹1000/month)
* PostgreSQL
* Redis
* Reverse proxy
* SSL certificate

Handles:

* All user-facing APIs
* ERP logic
* Authentication

---

## 7.2 Local AI Node (Data Plane)

Hardware:

* RTX 4090 recommended
* 64GB RAM
* 2TB NVMe
* UPS backup

Runs:

* LLM server
* Embedding model
* Vector DB
* Ingestion service

Exposed via secure tunnel to VPS.

No direct public IP exposure.
