---

# AI Engine Deep Design

**Project:** VidyaOS – AI Infrastructure for Educational Institutions
**Version:** v0.1 (Pilot → Cloud-Scale Ready)
**Deployment:** Local GPU (Pilot) → Distributed GPU Cluster (Scale)

---

# 1. Design Objectives

The AI Engine must:

1. Provide NotebookLM-level grounded document intelligence
2. Operate in strict multi-tenant isolation
3. Enforce citation-first outputs
4. Optimize GPU utilization
5. Remain cloud-migration ready
6. Never modify authoritative ERP records

AI is augmentation layer only.

---

# 2. Core Responsibilities

The AI Engine performs:

* Document ingestion
* Hierarchical chunking
* Embedding generation
* Vector storage
* Semantic retrieval
* Reranking
* Context compression
* LLM inference
* Structured output generation
* Citation enforcement

---

```

    ▼

    │
    ▼
Context Builder
    │
    ▼
Prompt Template Engine
    │
    ▼
LLM Inference
    │
    ▼
Response Formatter + Citation Injector
```

---

# 4. Multi-Tenant Isolation Model

Each institution has:

* Unique `tenant_id`
* Isolated vector namespace
* Independent document index
* Usage tracking

Vector DB structure:

```
namespace = f"tenant_{tenant_id}"
```

No cross-namespace retrieval permitted.

Hard constraint.

---

# 5. Document Ingestion Pipeline

## 5.1 Supported Sources

* PDF notes
* Lecture transcripts
* DOCX files
* Text uploads
* YouTube transcript ingestion

---

## 5.2 Preprocessing Pipeline

```
Raw Document
    │
    ▼
Text Extraction
    │
    ▼
Cleaning (remove headers, noise)
    │
    ▼
Hierarchical Segmentation
    │
    ▼
Metadata Attachment
    │
    ▼
Chunk Storage
```

---

## 5.3 Hierarchical Chunking Strategy

Flat chunking is insufficient.

Instead:

Document → Section → Subsection → Paragraph

Each chunk stores:

* tenant_id
* document_id
* chunk_id
* page_number
* section_title
* subject
* teacher_id
* academic_year

Chunk size:

* 300–600 tokens
* Overlap: 50–100 tokens

---

# 6. Embedding Engine

## Model

* nomic-embed-text (local)

## Flow

```
Chunk → Embedding Model → 768-dim Vector → Vector DB
```

Embeddings generated:

* On upload
* On transcript ingestion
* Async background job

Stored in tenant namespace.

---

# 7. Retrieval Pipeline

## 7.1 Query Flow

```
Student Query
    │
    ▼
Query Embedding
    │
    ▼
Vector Search (Top 8–12)
```

---

## 7.2 Reranking (Optional but Recommended)

Reranker:

* Cross-encoder model
* Scores semantic relevance

Select top 5–8 post-rerank.

This significantly improves answer quality.

---

## 7.3 Context Deduplication

Remove:

* Redundant chunks
* Near-identical overlaps
* Cross-page repetition

---

## 7.4 Context Compression

If context > LLM limit:

* Summarize lower-ranked chunks
* Keep highest semantic weight chunks raw

Goal:
Maintain relevance under token budget.

---

# 8. Prompt Template Engine

AI must not operate in free-form mode.

All responses pass through structured templates.

---

## 8.1 Task Modes

| Mode        | Description                 |
| ----------- | --------------------------- |
| Q&A         | Grounded question answering |
| Study Guide | Structured summary          |
| Quiz        | Question generation         |
| Concept Map | Graph structure             |
| Weak Topic  | Performance-based insight   |

---

## 8.2 Example Q&A Template

```
You are an academic assistant.
Answer ONLY using provided context.
Cite using [Document_Page].
If not in context, say: Not found in material.
```

This reduces hallucination risk.

---

# 9. LLM Inference Layer

## 9.1 Model Selection (Pilot)

* Qwen 14B (4-bit quantized)
* Llama 3 8B as fallback

---

## 9.2 Token Budget Strategy

Example:

* Query: 200 tokens
* Context: 3000 tokens
* Output: 600 tokens
* Total: ~3800 tokens

Must enforce strict caps.

---

## 9.3 GPU Utilization Strategy

* Queue-based request handling
* Max concurrency limit
* Timeout protection
* Batch embedding generation

---

# 10. Citation Enforcement Layer

Post-processing step:

1. Detect claim spans
2. Attach citation markers
3. Validate citation presence
4. Reject output if no citation (for Q&A mode)

Example output:

```
Photosynthesis occurs in chloroplasts. [Bio_Ch3_p12]
```

---

# 11. Structured Output Engine

For Quiz Mode:

Return JSON:

```
{
  "questions": [
    {
      "question": "...",
      "options": ["A","B","C","D"],
      "correct": "B"
    }
  ]
}
```

For Concept Map Mode:

Return graph schema:

```
{
  "nodes": [...],
  "edges": [...]
}
```

Frontend renders visualization.

---

# 12. Weak Topic Intelligence (Advanced)

Uses ERP data:

* Marks
* Subject performance
* Attendance

Flow:

```
Low Algebra Score
    │
    ▼
Fetch Algebra Notes
    │
    ▼
Generate Focused Study Guide
    │
    ▼
Generate Targeted Quiz
```

This is your differentiator over generic NotebookLM.

---

# 13. Caching Strategy

To reduce cost:

* Cache embeddings
* Cache top 20 frequent queries per subject
* Cache study guides per chapter
* TTL-based invalidation

Redis used for short-term caching.

---

# 14. Rate Limiting

Per student:

* X queries per day
* Tier-based AI credit system

Prevents GPU overload.

---

# 15. Observability

Track:

* Query latency
* Retrieval hit rate
* Average token usage
* GPU utilization
* Cache hit ratio
* Citation accuracy rate

Without metrics, scaling is blind.

---

# 16. Failure Handling

If:

* No relevant chunks found → return controlled fallback
* GPU overload → queue
* Timeout → retry
* AI hallucination detection triggered → reject output

Never return ungrounded answer silently.

---

# 17. Security Controls

* No external training
* Tenant namespace isolation
* Firewall-restricted AI server
* Internal API key validation
* No public model endpoint
